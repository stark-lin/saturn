# FILES Core Business Modules

> Split from `docs/FILES.md`: Business module common structure and core module file standards.

## 7. Business Module Common Structure

Business modules default to using the following structure:

```text
internal/{module}
├── dto.go
├── handler.go
├── handler_api.go
├── model.go
├── module.go
├── repo.go
├── repo_access.go
├── queries
│   └── *.sql
├── sqlc
│   └── *.go
├── service.go
├── query.go
├── jobs.go
└── service_test.go
```

### 7.1 File Responsibilities

| File | Responsibility |
| --- | --- |
| `dto.go` | Request and response structures |
| `handler.go` | HTTP handler common assembly |
| `handler_api.go` | API request binding and response |
| `model.go` | Domain models and database mapping structures |
| `module.go` | Module initialization and dependency injection |
| `repo.go` | Module repository interfaces and adaptation of generated queries to domain models |
| `repo_access.go` | Conversion of auth.Scope to fixed SQL helpers |
| `queries/*.sql` | sqlc named query templates for implemented database behaviors or existing repository contracts |
| `sqlc/*.go` | Type-safe database call code generated by sqlc |
| `service.go` | Business rules |
| `query.go` | List, filter, sort, and pagination parameters |
| `jobs.go` | Background task definitions and handlers for this module |
| `service_test.go` | Service layer tests |

### 7.2 Rules

The default structure is the recommended form for a complete module, not a mandatory one-time creation list. If a module has not implemented pages, background tasks, auth scopes, and has no defined and schema-backed repository contracts, do not create the corresponding empty files or empty generated packages.

It is allowed to add files by business behavior, for example:

```text
attachment.go
collection.go
event.go
file_link.go
note_link.go
subscription.go
summary.go
contributor.go
repo_access.go
repo_test.go
handler_test.go
validator.go
events.go
```

Adding fixed layering directories is prohibited:

```text
domain/
application/
infrastructure/
transport/
delivery/
usecase/
adapter/
```

Unless a single module is already obviously complex enough to necessitate splitting.

When a module becomes complex, prioritize splitting files by business behavior, such as `subscription.go`, `attachment.go`, `note_source.go`, `event_aggregate.go`. Do not stuff all business rules into a massive `service.go` just to maintain fixed layering, nor stuff all queries into a massive `repo.go`.

SQL Query Rules:

```text
Implemented PostgreSQL queries and persisted templates for existing repository contracts are written into this module's queries/*.sql, generating sqlc/*.go via sqlc.
repo.go maintains the domain adaptation boundary; the service does not directly depend on the sqlc generated package.
sqlc/*.go are committed to the repository but not edited by hand.
Generated templates do not imply that the handler, service, or repository adapter is assembled at runtime.
queries/ and sqlc/ are not created ahead of time for future behaviors lacking an existing repo contract.
```

### 7.3 Permissions and Audit Conventions

The business service defaults to receiving the current actor:

```go
func (s *Service) DeleteFile(ctx context.Context, actor auth.Principal, id int64) error
```

Conventions:

```text
handlers only act as authentication entries, parameter binding, and response writing
handlers do not perform resource-level permission checks
services are responsible for business rules, resource-level authorization, and critical operation audits
services only make authorization judgments via platform/auth
services only record audit events via platform/audit
repos do not redefine permission policies
repos can receive auth.Scope to filter lists and search results
```

## 13. `internal/platform` Product Capability Sub-packages

`internal/platform` simultaneously contains low-level support packages and product platform capability sub-packages. For complete details, see [PLATFORM.md](PLATFORM.md).

Current target sub-packages:

```text
internal/platform
├── audit
├── auth
├── config
├── db
├── httpx
├── logger
├── redis
├── ref
├── search
└── storage
```

Responsibilities:

```text
Audit: Audit event recording and querying
Auth: Login, session entry, identity and access control entry
Config: Configuration loading and runtime configuration exposure
DB: Database connection, migration execution, and transaction encapsulation
HTTPX: Request binding, response, error, and SSE encapsulation
Logger: structured logging encapsulation
Redis: Redis client setup used by auth sessions
ObjectRef: Unified readable object reference codes, title/status projection, registration, resolution, and owner-only metadata
Tag: Maintains tag associations for the same ObjectRef object that can be returned with metadata
Search: ObjectRef metadata HTTP handlers and compatibility search endpoint
Storage: local filesystem storage abstraction, usage, diagnostics, object reference management
```

Rules:

```text
ObjectRef/Tag aggregates ref_code, title, tags, status metadata, does not replace internal database ids, and does not bypass business services / facades
Storage does not own Files' business rules
Auth does not generate business SQL
Do not add empty platform packages for future scheduler, validation, queue, or indexing behavior
```
